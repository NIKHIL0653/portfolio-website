Understanding the Basics: Git vs. GitHub

First, let's clear up the biggest point of confusion: Git and GitHub are not the same thing. Think of it this way:

  • Git is the tool. It's a version control system that runs on your local computer. It's the software that tracks changes to your code, creates a history of those changes, and lets you manage different versions of your project.
  • GitHub is the platform. It's a website, a cloud-based service, that hosts your Git projects (called repositories). It provides a central place for you and your team to store your code and collaborate. It's the social network for developers.

In essence, you use Git on your machine to track your work, and you use GitHub to share that work with the world.

The First Two Steps: Install & Configure

Before you can do anything with Git, you need to set up your environment. This is a one-time process for any new computer you work on.

  1. Install Git: You need to have Git installed on your computer to use it from your command line or terminal. You can download the official Git software from git-scm.com.
  2. Set Up Your Local Credentials: Once Git is installed, you need to tell it who you are. This links your local commits to your GitHub account and is a critical first step. Open your terminal and run these two commands, replacing the information with your own details: 
    git config --global user.name "Your Name Here"
git config --global user.email "your.email@example.com"

    The --global flag tells Git to remember this information for every project you work on.

Getting Started: Your First Repository

A repository (or "repo" for short) is the core of any Git project. It's a folder on your computer that contains all your project's files, as well as a hidden folder where Git stores its version history.

Here's how to get your first project onto GitHub:

  1. Create a GitHub Account: If you haven't already, head over to the GitHub website and sign up. It's free and easy.
  2. Create a Repository on GitHub: Once you're logged in, click the New button on the top-left of the page to create a new repository. Give it a name (like hello-world), an optional description, and choose whether it should be public or private. I always recommend checking the box to "Initialize this repository with a README." This creates a file that you can use to describe your project.
  3. Clone the Repository to Your Computer: Now, you need to get a copy of that new empty repository onto your computer. Go to your new repo's page on GitHub, click the green <> Code button, and copy the URL. Then, open your terminal (or Git Bash on Windows), navigate to the folder where you want to store your projects, and run the following command: 
    git clone [paste the URL here]

    This will create a new folder on your computer with the same name as your repository. This is your local copy. When you run the git clone command for the first time, Git will prompt you for your GitHub username. For the password, do not use your GitHub password. Instead, you must use a Personal Access Token (PAT). GitHub has stopped supporting simple password authentication for security reasons.

How to get a Personal Access Token:

  1. On GitHub, click your profile picture, then go to Settings.
  2. In the left sidebar, click Developer settings.
  3. Click Personal access tokens, then Tokens (classic).
  4. Click Generate new token. Give it a descriptive name, set an expiration, and check the repo scope.
  5. Click Generate token at the bottom. Copy this token immediately and save it somewhere secure. You won't be able to see it again.

When Git prompts you for a password, you'll paste this token.

Making Your First Change

Now that you have your local repository, you can start working on it.

  1. Add a New File: Inside your new folder, create a new file, for example, index.html.
  2. Stage Your Changes: Git needs to know which changes you want to save. This is called staging. Open your terminal in your project folder and run: 
    git add index.html

    Or, to add all new files and changes:

    git add .
  3. Commit Your Changes: A commit is like a snapshot of your project at a specific moment in time. It's a way of saying, "Okay, I'm done with this set of changes, let's save them." Every commit needs a descriptive message. Run the following command: 
    git commit -m "Add initial index.html file"
  4. Push to GitHub: So far, all your work is only on your local machine. The last step is to push those changes to the remote repository on GitHub. This makes your work visible to others (or saves it in the cloud for you). 
    git push

And that's it! You've successfully made your first commit and pushed it to GitHub. You can now refresh your repository page on GitHub and see your new file.

A Common Scenario: Pushing an Existing Project to GitHub

What if you've already started a project on your computer and now want to put it on GitHub? Don't worry, you don't have to start from scratch.

Here's how I did it with a project I was already working on:

  1. Create an Empty GitHub Repository: The first step is to create a new, empty repository on GitHub, just like before. This time, don't check the box to "Initialize this repository with a README."
  2. Initialize Git in Your Project Folder: Open your terminal and navigate to your project folder. Now, run this command: 
    git init

    This command initializes a new, empty Git repository in your current directory. You've just told Git, "Start tracking this folder for me."

  3. Link Your Local Repo to the GitHub Repo: Now you need to tell your local Git project where to find its remote home on GitHub. Go to your newly created, empty repository on GitHub and copy the URL. Then, back in your terminal, run: 
    git remote add origin [paste the URL here]

    The git remote add origin command essentially gives your GitHub repository a nickname (origin) so you don't have to use the full URL every time.

  4. Add, Commit, and Push Your Files: The rest of the process is the same as before. You'll add all your existing files, make an initial commit, and then push them to GitHub. 
    git add .
git commit -m "Initial commit of existing project"
git push -u origin main

    The -u origin main flag is a special one. It sets up the relationship between your local main branch and the remote main branch, so for all future pushes, you can just use git push.

And that's it! You've now taken a project that existed only on your computer and given it a safe, collaborative home on GitHub.

Even with these steps, you might run into some common errors. Don't panic! Here are a few I've encountered and how I solved them.

1. remote: Repository not found

This error usually means one of two things:

  • The repository name or URL is wrong.
  • You don't have the permissions to access it.

How to fix:

Double-check the URL you're using. If it's a private repository, make sure your GitHub account is a collaborator and that you've used a valid PAT when prompted.

2. fatal: not a git repository (or any of the parent directories): .git

This happens when you run a Git command outside of a folder that has been initialized as a Git repository.

How to fix:

Navigate to the correct project folder in your terminal and then run the command again. If it's a new project, remember to run git init first.

3. failed to push some refs to... (non-fast-forward error)

This is one of the most common errors for beginners. It means someone else has pushed new changes to the remote repository, and your local copy is now out of date. Git won't let you push your changes because it would overwrite their work.

How to fix:

You need to get the latest changes from the remote repository before you can push your own. The simplest way is to run:

git pull origin main

This will pull the new changes from GitHub and merge them with your local changes. Once the merge is complete, you can try your git push command again.
